1. ARCHITECTURAL OVERVIEW
=========================

The Grevya Naturals application is structured as a modern, decoupled, client-first architecture:

* Primary Frontend Stack: React (v18.3.1), Vite (v5.4.1), TypeScript, TailwindCSS (v3.4.11), Framer Motion (v12.34.4).
* Core Backend / Database: Supabase (PostgreSQL, Auth, Storage).
* Legacy Backend: An Express/MongoDB stack exists inside the `/server` directory, but is currently unused by the frontend codebase.

Client-Side Routing
-------------------
Managed client-side in [App.tsx](file:///f:/Grevya/Grevya---ECommerce/src/App.tsx) using React Router DOM's `<BrowserRouter>`.
* Includes a catch-all route (`<Route path="*" element={<NotFound />} />`) for graceful 404 handling.
* Protected views (e.g., `/account`, `/orders`) are wrapped in the `<ProtectedRoute>` component.
* Production URL rewriting is configured via [vercel.json](file:///f:/Grevya/Grevya---ECommerce/vercel.json) to redirect all routes to `index.html` to prevent Vercel server-side 404s on page refreshes.

State Management
----------------
* Zustand: Used for transient, local storage-persisted user state:
  - [useCartStore.ts](file:///f:/Grevya/Grevya---ECommerce/src/store/useCartStore.ts): Persists user cart items.
  - [useWishlistStore.ts](file:///f:/Grevya/Grevya---ECommerce/src/store/useWishlistStore.ts): Persists user wishlist items.
* React Context:
  - [AuthContext.tsx](file:///f:/Grevya/Grevya---ECommerce/src/context/AuthContext.tsx): Manages auth session state, profile loading, and profile synchronization.

3. AUTHENTICATION & SESSION LIFECYCLE
=====================================

Anonymous Guest Auth & Promotion
--------------------------------
1. Guest Checkout Flow: If an unauthenticated visitor clicks "Checkout", [Checkout.tsx](file:///f:/Grevya/Grevya---ECommerce/src/pages/Checkout.tsx) automatically triggers `supabase.auth.signInAnonymously()` to log them in as an anonymous user.
2. Order Association: Guest orders and shipping addresses are recorded under this anonymous UUID, eliminating the need for separate unauthenticated tables.
3. Promotion Flow: If the anonymous user registers an account on [AuthPage.tsx](file:///f:/Grevya/Grevya---ECommerce/src/pages/AuthPage.tsx), `signUpWithEmail` invokes `supabase.auth.updateUser` with the password and new email.
4. Data Persistence: The user's UUID is preserved during promotion. This seamlessly inherits all past orders, addresses, and notifications onto their newly registered account.

Auth Concurrency & Anti-Spam Protections
-----------------------------------------
[authService.ts](file:///f:/Grevya/Grevya---ECommerce/src/lib/authService.ts) enforces:
* Concurrency Lock: A `locks` map tracks pending promises by action key, preventing concurrent duplicate clicks/requests from firing.
* Rate Limit Cooldowns: An active `cooldowns` map plus local storage timestamps block spamming endpoints:
  - Standard sign up: 90 seconds.
  - Password recovery email: 60 seconds.
  - Phone OTP request: 60 seconds.
  - OAuth initiation: 5 seconds.

Session Synchronization
-----------------------
* `AuthContext` listens to `onAuthStateChange`. When a user logs in:
  - Zustand stores (`useCartStore`, `useWishlistStore`) run `syncUserSession(userId)`.
  - If transitioning from a guest, the store merges guest items into the user's persisted array and flushes the `'guest'` namespace in local storage.
  - `ensureUserProfile` in [profileSync.ts](file:///f:/Grevya/Grevya---ECommerce/src/lib/profileSync.ts) verifies/upserts the profile row in `public.profiles`.

4. DATABASE MODEL & SUPABASE TRIGGERS
=====================================

The database schema utilizes row-level security (RLS) policies and PostgreSQL sync triggers to handle code discrepancies and schema drift.

Triggers
--------
* `on_auth_user_created` (executes `public.handle_new_user()`):
  Fires when a user registers. Automatically upserts metadata into `public.profiles`.
* `trg_sync_address_codes` (executes `public.sync_pincode_and_postal_code()`):
  Fires before insert/update on `public.addresses`. Ensures `pincode`/`postal_code`, `address_line1`/`address_line_1`, and `address_line2`/`address_line_2` are identical, resolving schema name mismatch errors.
* `trg_sync_order_totals` (executes `public.sync_order_totals()`):
  Fires before insert/update on `public.orders`. Syncs `total`/`total_amount` and `status`/`order_status`. Generates a tracking number (`TRK-...`) if null.
* `trg_sync_notification_read` (executes `public.sync_notification_read_flags()`):
  Fires before insert/update on `public.notifications`. Syncs `read` and `is_read`.
* `trg_normalize_user_email_change_tokens` (executes `public.normalize_user_email_change_tokens()`):
  Fires before insert/update on `auth.users` to normalize token fields to empty strings, preventing GoTrue SMTP crashes.

Security Hardening & RLS Models
--------------------------------
* RPC Exposure: Executable privileges on all custom definer functions are revoked from `public`, `anon`, and `authenticated` roles. Trigger executions bypass these checks, but API REST RPC queries are blocked.
* Reviews Table: RLS only permits `authenticated` users to insert reviews under their own `auth.uid()`.
* Storage Buckets (`profile-images` and `avatars`): SELECT policies are hardened with `NOT storage.allow_only_operation('object.list')`. This blocks listing files via API calls while preserving direct render/downloads via public URLs.

5. E-COMMERCE LIFECYCLE
=======================

Order Submission Resilience
---------------------------
To prevent database changes from breaking active checkouts, [Checkout.tsx](file:///f:/Grevya/Grevya---ECommerce/src/pages/Checkout.tsx) wraps order inserts in a retry loop:
* If a column fails to insert because it does not exist (e.g., column cache drift), `safeInsertOrder` intercepts the error, deletes that field from the payload, prints a warning to console, and retries.

Payment Integration
-------------------
Payments are handled in [razorpay.ts](file:///f:/Grevya/Grevya---ECommerce/src/lib/razorpay.ts):
* Production: Loads the external Razorpay SDK and opens the payment modal.
* Local Development (`import.meta.env.DEV`): If Razorpay credentials are missing, falls back to a simulated success prompt via `window.confirm`.

6. FUTURE DEVELOPMENT GUIDELINES
================================

To maintain codebase sanity, preserve these patterns:

* Database Syncing: Always write queries to query both sync fields (e.g. `total` and `total_amount`) or let database triggers handle normalization.
* Guest State Transition: Do not store unauthenticated cart/wishlist items under user UUIDs before login; rely on Zustand's `syncUserSession` to merge them cleanly.
* Resilient Queries: Use the "safe" DB wrappers (`safeInsertOrder`, `safeInsertOrderItems`) for core checkout operations.
* File Conventions:
  - PascalCase for React components and pages.
  - camelCase for hooks, services, and utility functions.
  - snake_case for database columns.
  - CSS tokens in [index.css](file:///f:/Grevya/Grevya---ECommerce/src/index.css) must be used instead of custom ad-hoc Tailwind classes to preserve the premium brand aesthetic.
